// PROJECT CASE STUDY

Elision Analytics
Dashboard.

A proof of concept analytics dashboard built for a real client — Elision needed a way to track real page load times across different regions and devices. Google Analytics wasn't giving them accurate data, so we built a custom tracking solution with a React dashboard and a lightweight SQL backend, in a team of 6 using Agile and Scrum.

// 01 · OVERVIEW

Overview.

01 / PROBLEM

Elision had no reliable way to measure how long their pages actually took to load for real users. Google Analytics wasn't providing accurate load time data across different regions and devices, making it impossible to identify performance bottlenecks or prioritise improvements.

02 / ASSIGNMENT

Build a custom analytics solution for a real client as a proof of concept. The system needed a tracking script that could be embedded in any page to capture real load times, combined with a dashboard to visualise the data by region, device and page.

03 / SOLUTION

A lightweight JavaScript tracking script captures page load times and sends them to a Java Spring Boot API. The data is stored in a lightweight SQL database and visualised in a React dashboard — showing load times per region, device type and page, with filtering and comparison tools.

// 02 · TECHNICAL ARCHITECTURE

Architecture.

Tracking Script
JavaScript · Embeddable

A lightweight JavaScript snippet that can be embedded in any website. It captures the browser's performance timing data — specifically the actual page load time as experienced by the user — and sends it to the API along with the user's approximate region and device type.

Java Spring Boot API
Java · Spring Boot · REST

The REST API receives load time events from the tracking script, validates and enriches the data, and writes it to the database. Built with Java and Spring Boot, it exposes endpoints for both ingesting new events and querying aggregated analytics data for the dashboard.

Lightweight SQL Database
SQLite

Load time events are stored in a lightweight SQL database. The schema is optimised for the analytics queries the dashboard needs — aggregations by page, region and device — keeping query performance fast even as the event volume grows.

React Dashboard
React · Tailwind CSS

The React frontend fetches aggregated data from the API and visualises it in an interactive dashboard. Users can filter by page, region and time period, and compare load times across different segments. Built with Tailwind CSS, designed in Figma first.

// 04 · CHALLENGES

Challenges.

01 / SLOW CLIENT REVIEWS

The client ran their own code reviews, which were thorough but slow to come back. Waiting for feedback blocked certain parts of the project from moving forward, creating bottlenecks in the sprint.

→ Approach

Learned to plan around review cycles rather than against them. When a review was pending, the team switched to other tasks instead of waiting. This kept sprint velocity consistent and reduced the pressure when reviews finally came back.

02 / DIFFICULT EARLY MEETINGS

The first few client meetings were difficult — expectations weren't clearly aligned, the client's feedback was sometimes hard to translate into actionable tasks and the team struggled to get clear answers on scope.

→ Approach

We changed our approach — instead of open-ended discussions, we came to meetings with working prototypes and specific questions. Showing something concrete made it much easier for the client to give useful feedback and for us to lock down scope.

03 / TEAM COORDINATION UNDER PRESSURE

As the deadline approached and review bottlenecks accumulated, keeping the team focused and motivated became a challenge in itself. Different parts of the codebase were at different stages of completion.

→ Approach

Taking on the team lead role gave me a clear mandate to coordinate. I made a simple task board showing what was blocked, in progress and done — giving everyone visibility and making it obvious where help was needed.

04 / DESIGN TO CODE

Translating a Figma design into a working React implementation — especially for data visualisation components — required careful attention to responsive behaviour and state management that wasn't always obvious from the static design.

→ Approach

Built components incrementally — starting with static versions that matched the design, then layering in real data and interactive behaviour. This made it easier to spot layout issues early before adding complexity.
