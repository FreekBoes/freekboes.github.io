// PROJECT CASE STUDY

Handbal
Overpelt.

A full redesign and rebuild of the public website for Handbal Overpelt — my local handball club. The old site was outdated and no longer represented the club well. I took the initiative, rebuilt it from scratch and deployed it on their existing hosting.

// 01 · OVERVIEW

Overview.

01 / PROBLEM

The club's existing website was outdated — visually dated, not mobile-friendly and missing content that members and visitors actually needed. It no longer reflected the club's energy or gave a good first impression to potential new members.

02 / ASSIGNMENT

Self-initiated: analyse what the old site was missing, decide what to keep and what to cut, and build a modern, responsive replacement that works on any device and is easy for the club to maintain going forward.

03 / SOLUTION

A fully responsive static website built with HTML, Tailwind CSS and JavaScript. The new site covers all teams, the club calendar, sponsor information, activities and contact — deployed directly on the club's existing hosting platform.

// 02 · ARCHITECTURE

Architecture.

HTML · Tailwind CSS · JavaScript
Static · No backend

The site is built as a static website — no framework, no backend, no database. HTML structures the content, Tailwind CSS handles all styling through utility classes, and vanilla JavaScript adds interactive elements like the mobile navigation and dynamic content.

Hosting & migration
Existing hosting · Zero downtime

Deployed directly on the club's existing hosting platform, keeping costs at zero. The migration from old to new was handled without downtime — the domain stayed live throughout the transition.

Version control
GitHub

The project is version-controlled on GitHub, giving the club a recoverable codebase and making it straightforward to push updates when content changes.

// 04 · CHALLENGES

Challenges.

01 / NO BRIEF, NO CLIENT DIRECTION

This project was entirely self-initiated. There was no brief, no requirements document and no one telling me what the site needed. Every decision — what to include, what to cut, how to structure it — came from me.

→ Approach

Treated it as a real client project from the start. Did a content audit of the old site, mapped out the pages and structure before touching any code, and made decisions based on what a club member or new visitor would actually need.

02 / CONTENT DECISIONS

The old site had accumulated content over years — some still useful, some outdated, some just cluttering the navigation. Deciding what to migrate and what to leave behind required judgement, not just execution.

→ Approach

Kept only what was either actively useful or important for the club's image. Cut everything that would require constant maintenance or was no longer accurate.

03 / MIGRATION WITHOUT DOWNTIME

The club's domain was live and in use during the rebuild. Switching from the old site to the new one had to happen cleanly — no broken links, no period where the site was unavailable.

→ Approach

Built and tested the full new site locally before touching the hosting. Deployed in a single planned switch, verifying all pages and links immediately after going live.

// 05 · WHAT I LEARNED

What I Learned.

// SOLO CLIENT OWNERSHIP

Running a project from first idea to live site without a team or manager taught me that the decisions no one else makes are yours. Scope, content, structure, timing — all of it. That's a different kind of responsibility than contributing to a shared project.

// CONTENT IS A DESIGN DECISION

Deciding what goes on a website is as important as how it looks. Cutting content that doesn't serve the visitor and structuring information so it's easy to find are design problems, not just editorial ones.

// SHIPPING AND MAINTAINING

Launching a site is not the end of the project. Ongoing maintenance — content updates, fixes, keeping things current — is part of the contract. That relationship with a live product teaches you things a finished school project never does.
