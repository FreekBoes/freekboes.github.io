// PROJECT CASE STUDY

Pulse.

A personal Flutter app for tracking gym workouts — built because the features I wanted were locked behind a paywall. Plan sessions, track progress and analyse your training, all stored locally on device.

Overview Features Architecture My Contribution Challenges What I Learned
TECH Flutter Dart GitHub
// 01 · OVERVIEW

Overview.

01 / PROBLEM

Existing workout tracking apps put the features I actually wanted — 1RM calculation, workout history, custom schemas — behind subscriptions. I wanted a single app that covered everything I needed, free and offline.

02 / ASSIGNMENT

Self-directed: build a mobile gym tracking app. Set up the project plan, draw up a roadmap, and ship a working product on my own timeline — built after hours during my internship.

03 / SOLUTION

A Flutter app covering the full training workflow: 1RM estimation, session tracking, reusable workout schemas, a full training history, and more — all stored locally on device with no subscription required.

// 02 · FEATURES

Features.

🏋️
1RM calculator

Estimate your one-rep max for any exercise based on your working sets — no guessing required.

📋
Workout tracking

Log every set, rep and weight for every session. Track progress over time across all exercises.

🗂️
Workout schemas

Create reusable workout plans and load them when you start a session — no re-entering the same exercises every time.

📈
Training history

A full log of every session: when you trained, what you did and how your lifts have progressed.

📅
Session frequency

See how often you have been to the gym and spot patterns in your training consistency.

⚡
More

Additional features built around the core workflow — everything you need, nothing you don't.

// 03 · ARCHITECTURE

Architecture.

Flutter Frontend
Flutter · Dart

The entire app is built with Flutter, delivering a native-feeling UI on Android and iOS from a single Dart codebase. All business logic, state management and UI rendering run in Dart.

Local storage
On-device · No backend

All training data is stored locally on the device. No external database or server — the app works fully offline and your data stays private by default.

Version control
GitHub

The project is version-controlled on GitHub, with commits following a structured workflow based on the roadmap defined at the start of the project.

// 05 · CHALLENGES

Challenges.

01 / TIME MANAGEMENT

This project was built after hours during my internship. Available time was limited and had to be managed carefully alongside full working days.

→ Approach

The upfront roadmap kept scope in check. Working in priority order meant the core features shipped first — the app was always in a usable state even when time was short.

02 / SCOPING THE FEATURE SET

With no external brief or team to push back, it was easy to keep adding features. A gym tracking app has an almost unlimited potential scope — discipline was required.

→ Approach

Anything not on the roadmap did not get built until the core was done. The roadmap acted as the contract with myself.

03 / CODE QUALITY SOLO

Without a team or code reviews, maintaining quality standards purely through self-review required conscious effort — especially when tired after a full day of internship work.

→ Approach

Applied the same review habits I used in team projects: reviewing my own PRs before merging, keeping commits small and descriptive, and refactoring before adding new features.
