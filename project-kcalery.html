// PROJECT CASE STUDY

Kcalery.

A personal Flutter app for tracking calories, macros, water intake and creatine — built because existing apps either lacked the features I needed or locked them behind a paywall.

Overview Features Architecture My Contribution Challenges What I Learned
TECH Flutter Dart GitHub
// 01 · OVERVIEW

Overview.

01 / PROBLEM

Most nutrition apps either miss features I wanted — like a creatine tracker or full macro breakdowns — or hide them behind subscriptions. I wanted one app that did exactly what I needed, nothing more.

02 / ASSIGNMENT

Self-directed: build a mobile app for daily nutrition tracking. Define the feature set, plan the architecture, and ship a working product on my own timeline — after returning from internship.

03 / SOLUTION

A Flutter app with calorie and macro tracking, water intake logging, a creatine check-in, a diet mode selector and a profile system that calculates your daily targets — all stored locally on device.

// 02 · FEATURES

Features.

🔥
Calorie tracking

Log meals and track daily calorie intake against your personalised target.

⚖️
Macro breakdown

Full insight into protein, fat and carbohydrate intake per day.

💧
Water tracker

Log water intake throughout the day and track progress towards your hydration goal.

💊
Creatine tracker

A simple daily check-in: did you take your creatine today? No subscription required.

🥗
Diet mode

Choose a diet approach — high protein, low carb, balanced — and the app adjusts your macro targets accordingly.

👤
Profile & targets

Set your personal details and let the app calculate how much you should eat per day based on your goals.

// 03 · ARCHITECTURE

Architecture.

Flutter Frontend
Flutter · Dart

The entire app is built with Flutter, giving a native-feeling UI on Android and iOS from a single codebase. Dart handles all business logic, state management and UI rendering.

Local storage
On-device · No backend

All data is stored locally on the device. There is no external database or server — the app works fully offline and keeps your data private by default.

Version control
GitHub

The project is version-controlled on GitHub. Commits follow a structured workflow based on the roadmap defined at the start of the project.

// 05 · CHALLENGES

Challenges.

01 / SCOPING THE FEATURE SET

With no external brief or team to push back, it was easy to keep adding features. Deciding what was in scope and what wasn't required discipline.

→ Approach

The upfront roadmap acted as the contract with myself — if a feature wasn't on it, it didn't get built until the core was done.

02 / TIME MANAGEMENT

This project was built after returning from internship, meaning available time was limited and had to be managed carefully alongside other commitments.

→ Approach

Working from the roadmap in priority order meant the most important features shipped first, so the app was always in a usable state.

// 07 · WHAT I LEARNED

What I Learned.

// PROJECT MANAGEMENT
