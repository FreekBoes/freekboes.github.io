// PROJECT CASE STUDY

De Beerse
Bende.

A custom web application built for a real client — a local entertainment group that needed to digitise their entire show operation. Ticket sales, accounting and scheduling were all done on paper. We replaced that with a full-stack web platform, built in a team of 6 using Agile and Scrum over one semester.

// 01 · OVERVIEW

Overview.

01 / PROBLEM

The client managed everything manually — tickets were issued on paper, show accounting was entered by hand and there was no central overview of revenue, attendance or scheduling. This made the operation slow, error-prone and hard to scale.

02 / ASSIGNMENT

Digitise the client's entire show operation within one semester. Build a custom web platform that handles ticket sales, financial accounting and show scheduling — delivered as a working product to a real client, not a mock exercise.

03 / SOLUTION

A full-stack web application built with Laravel, Livewire and Tailwind CSS. The platform covers the complete show operation — ticket management, financial reporting, a calendar for scheduling and role-based access for different user types.

// 02 · TECHNICAL ARCHITECTURE

Architecture.

Frontend — Livewire & Tailwind CSS
Laravel Livewire · Tailwind CSS

The user interface is built with Laravel Livewire, which allows reactive, dynamic UI components without writing separate JavaScript. Tailwind CSS handles all styling. The result is a responsive, modern interface that updates in real time when data changes.

Backend — Laravel & PHP
PHP · Laravel

The backend is built with Laravel following MVC conventions. It handles all business logic — ticket processing, financial calculations, access control and calendar management. PHP powers the server-side logic, with Laravel's routing and service layer keeping the codebase maintainable across a team of 6.

API Layer
REST · Laravel

A REST API layer exposes the platform's data in a structured way. This keeps the frontend and backend cleanly separated and makes it possible to extend the platform with additional interfaces — for example a mobile app — without rewriting the backend logic.

Database — Migrations & Seeders
MySQL · Laravel migrations

The database is managed through Laravel migrations and seeders, keeping the schema version-controlled alongside the codebase. The data model covers shows, tickets, users, roles, financial records and calendar events — designed to be consistent and maintainable across the whole team.

// 04 · CHALLENGES

Challenges.

01 / WORKING IN A TEAM OF 6

Coordinating six developers on a single codebase means merge conflicts, overlapping work and different coding styles. Without clear agreements and structure, quality drops fast.

→ Approach

We set up a clear branching strategy on GitHub and used pull requests with mandatory code reviews before any merge. This kept the main branch clean and gave everyone visibility on what others were building.

02 / TRANSLATING CLIENT NEEDS

The client knew what they wanted in practical terms but not in technical terms. Bridging that gap — understanding their workflow and translating it into data models and features — was harder than the implementation itself.

→ Approach

We ran regular client meetings where we showed working prototypes and asked specific questions about their real workflow. Seeing the application in action gave the client a concrete way to give feedback instead of abstract requirements.

03 / ALIGNING ON DECISIONS

Six people means six opinions. Architectural choices, naming conventions and feature scope were all points where disagreements could slow the team down if not resolved quickly.

→ Approach

We used our sprint retrospectives to surface disagreements and resolve them as a team. When decisions had to be made quickly, we defaulted to what the majority agreed on and documented it so everyone was on the same page.

04 / CALENDAR COMPLEXITY

Building the calendar module meant handling recurring events, show slots, conflicts and a data model that had to stay consistent with ticket and financial data. It was the most technically complex single feature in the project.

→ Approach

Designed the calendar data model first before writing any backend logic — mapping out all the edge cases on paper before touching the code. This saved significant rework later and gave a clear contract for the Livewire frontend to work against.
